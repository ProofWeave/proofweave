## [슬라이드 2: 문제 정의] (1분 30초)

먼저 저희가 풀려는 문제입니다.

요즘 ChatGPT, Claude 같은 AI가 보고서, 분석 자료를 엄청 많이 만들어내잖아요?
근데 문제는, 이 데이터를 **"누가 만들었고, 중간에 조작은 안 된 건지"** 확인할 방법이 없다는 겁니다.

그리고 하나 더. AI 에이전트들이 점점 자동화되고 있는데, 
예를 들어 보안 분석 에이전트가 "Aave 취약점 분석"이 필요하다고 합시다.
지금은 직접 크롤링하고 정리해야 되는데, 이게 **토큰 5만 개, 돈으로 500원** 정도 듭니다.
근데 누군가 이미 정리해둔 걸 **50원에 사면?** 10분의 1 비용이죠.
그런데 이런 **"에이전트 간 데이터 거래"** 인프라가 아직 없습니다.

기존에 비슷한 것들이 있긴 합니다.
**EAS**라는 오픈소스는 온체인 attestation — 뭔가를 증명하는 기록을 남기는 건 잘 하는데, **결제 기능이 없어요.**
**x402**라는 Coinbase가 만든 오픈소스는 결제를 잘 하는데, **데이터 출처 증명이 없어요.**

저희 **ProofWeave**는 이 두 가지를 합친 겁니다. **출처 증명 + 결제 통합.**

초기 타깃 도메인은 **DeFi 보안 리서치**입니다.

## [슬라이드 4: 아키텍처] (1분 30초)

이제 시스템 구조를 보겠습니다.

크게 4층으로 되어 있는데요,

**맨 위: 사용자 계층.** 
처음에는 **CLI**, 그러니까 터미널에서 명령어를 치는 방식으로 데모를 만들고요,
나중에 Phase 2에서는 **웹이나 앱 서비스**로 전환합니다.
이때 웹에서 **GPT-4o, Claude 같은 LLM API를 연동**해서 데이터를 생성하고,
생성된 데이터를 **저희 ProofWeave 서비스에 등록**하는 구조입니다.
다른 사용자가 정보를 찾으려면 **ProofWeave를 통해서 검색**해야 해요. 외부에서는 못 찾습니다.

**두 번째: API 서버.** 
Express로 만든 TypeScript 백엔드입니다.
왜 API 서버가 필요하냐면, ChatGPT나 Claude 같은 에이전트들은 **HTTP 호출은 할 수 있지만, 블록체인에 직접 트랜잭션을 보내는 건 못합니다.** 
그래서 **API가 대신** 블록체인에 기록해주고, 결제도 대행해주는 구조입니다.

**세 번째: Rust 유틸리티.** 
JSON 해싱이랑 암호화를 Rust로 만든 CLI 도구가 처리합니다.
왜 Rust냐면, 해시값이 1바이트만 달라도 전체 검증이 깨지거든요.
이런 보안 민감한 영역은 **메모리 안전한 언어**가 필요합니다.

**맨 아래: 데이터 계층.** 
PostgreSQL은 검색용, IPFS는 원본 데이터 저장, Base Sepolia 블록체인은 증명 기록.

## [슬라이드 5: 데이터 흐름] (1분)

흐름을 간단히 보면요,

**생산자** Agent A가 보안 보고서를 만들어서 API에 보냅니다.
API가 JSON을 정규화하고 SHA-256 해시를 만들어요. 
해시는 — 데이터의 지문이라고 생각하시면 됩니다.
이 해시를 블록체인에 기록하고, 원본은 IPFS에 저장합니다.

**소비자** Agent B가 "DeFi 보안 자료" 검색 → ProofWeave에서 목록 확인 →
해시값 비교로 위변조 안 됐는지 검증 →
유료면 결제 → 데이터 받아서 사용.

핵심은, **모든 검색과 거래가 ProofWeave API를 통해서만** 이루어진다는 겁니다.

## [슬라이드 6: 오픈소스] (45초)

활용한 오픈소스를 정리하면,

**참조한 프로토콜**이 2가지 있는데요,
**EAS** — Ethereum Attestation Service — 온체인에 기록을 남기는 설계 패턴을 참고했고요,
**x402** — Coinbase가 만든 HTTP 결제 표준 — 결제 흐름 설계를 참고했습니다.
이 두 프로젝트에서 **필요한 개념만 가져와서 합친 것**이 저희 ProofWeave입니다.

직접 가져다 쓰는 오픈소스는,
**OpenZeppelin** — 스마트 컨트랙트 보안 표준 라이브러리. 프록시 패턴 등을 가져왔고,
**Foundry** — 스마트 컨트랙트 빌드/테스트 도구. Hardhat보다 10배 빠릅니다.

## [슬라이드 7: 기술 스택] (45초)

기술 선택 근거를 간단히 짚으면,

**왜 Base 체인?** → L2라서 가스비가 메인넷 대비 100배 저렴합니다.
**왜 UUPS 프록시?** → 스마트 컨트랙트는 한번 배포하면 수정이 안 되는데, 프록시 패턴으로 업그레이드가 가능하게 했습니다.
**왜 viem?** → 블록체인 연동 라이브러리인데, ethers.js보다 타입 안전성이 높습니다.
**왜 IPFS?** → 분산 저장이라 서버 하나 죽어도 데이터가 살아있고, CID 자체가 해시값이라 이중으로 무결성을 보장합니다.
**왜 온체인/오프체인 분리?** → 블록체인에 데이터 1KB 올리면 수천 원이에요. 그래서 **해시(지문)만 온체인에, 원본은 오프체인에** 저장합니다.

## [슬라이드 9: 예상 문제점] (1분)

마지막으로 예상 문제점입니다. **악의적 사용자가 나쁜 데이터를 넣으면 어떡하나?**

4가지 시나리오를 준비했는데요,

**첫째, 가짜 보고서를 올리는 경우.** 
→ API Key 발급 시 지갑 서명 인증을 해야 해서 익명 등록이 안 되고, 
블록체인에 creator 주소가 영구 기록되니까 추적이 가능합니다.

**둘째, 쓰레기 데이터를 대량으로 올리는 스팸.**
→ 등록마다 가스비가 들어서 자연스럽게 비용 장벽이 되고, API에 Rate Limit도 걸어둡니다.

**셋째, 악성 코드를 심는 경우.**
→ 데이터는 IPFS에 파일로만 저장되고 실행되지 않습니다. API가 사이즈 제한, 포맷 체크도 합니다.

**넷째, 돈 받고 쓰레기를 주는 경우.**
→ 여기서 중요한 건, 저희 프로토콜은 **"위변조 안 됐다"는 무결성만 보장**하고, 
**"내용이 좋다"는 품질은 등록자 책임**입니다. 이건 의도적 설계예요.
향후에 신고/환불 정책이랑 creator별 신뢰 점수를 도입할 예정입니다.
